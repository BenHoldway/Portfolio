Those Things From Below is a hoard-shooter survival first-person shooter, where you must activate and use the factory that you found yourself within to aid your fight against hoards of creatures.

From making use of different steam powered weapons and traps, you need to power up generators to get power to the factory. Use this power to progress through the game and escape from the hoard.

This was the 1st part of a 2 year module, in which I worked within a group at an industry level development module. Once the project was designed, which I had a part in, and had been pitched to the lecturers by the Senior team, I was tasked with creating and maintaining the Player Systems.

Project Development

To get the player into the level and working, I created a very simple system which automatically creates an owning hierarchy from the Player Controller and Player Pawn, which are both instantiated by the Game Mode Actor. Once this was created, and a simple First Person Pawn Blueprint was added to the project, I informed the team that they would be able to use this to test their systems as well.

With the basic Player implemented, I was able to create further systems like sprinting, crouching, interacting and a somewhat complex knee slide.

Through the rest of development, I worked within the Player Pawn Blueprint, as well as the Player Component Blueprints (like the Stamina and Powerup Timer Handler Components). I worked very closely with others who were developing systems close to the player, like the UI and pickup items. This was so that I was able to properly and efficiently set up communication to these systems from the player.

This project was my first full group development project that I had worked in and was able to learn a lot from it. While I know that I was fully engaged with this project and actively worked with others to get tasks done, which was backed by the feedback I received, there was a lot of improvements I was able to recognise.

One such thing that I learnt was how important communication is. The development process as a whole was a mess, with teams not understanding others and information being lost. This caused a lot of re-imaginings with design and changes happening within the very short development cycle. With better communication across the team, and with myself, the development process would have been a lot smoother.
